Context

SüdWest Sanierung (SWS) is a German property-damage restoration company (water, fire, mold, storm) that currently runs on Trello boards + Softr/Airtable/Make. They need two interconnected systems built from scratch on a custom stack (Next.js 14, Supabase, tRPC, n8n):



Schadenradar — a multi-tenant SaaS portal where external partners (Hausverwaltungen, Versicherungen, Makler, etc.) can submit and track damage claims in real time

SWS ERP — an internal operations system covering 20+ modules (project management, offers, invoices, disposition, inventory, HR, fleet, etc.)



Developer: Solo + Claude Code. Hosting: EU-preferred (Vercel EU + Supabase EU). Domain: schadenradar.de (owned), ERP domain TBD.



1. Framework Inconsistencies Found

These must be resolved before coding begins:

1.1 Role Model Conflict (CRITICAL)



A1 domain model defines: SYSTEM_ADMIN, PROJEKTLEITER, INNENDIENST, MULTIPLIKATOR_ADMIN, MULTIPLIKATOR_USER, SACHVERSTAENDIGER

Schadenradar Framework defines: SYSTEM_ADMIN, TENANT_ADMIN, MEMBER, VIEWER, REVIEWER, INTEGRATION

A2 DB schema uses the A1 roles in SQL ENUMs

Decision: Use Schadenradar Framework roles (per user input). The A2 SQL ENUMs and all RLS policies must be rewritten to match.



1.2 RBAC Policy Conflicts



Framework Section B has claim_reviewers table referenced in RLS but never defined in A2 schema

A2 schema has claim_assignments with claim_assignment_role ENUM but Framework uses claim_reviewers

Fix: Add claim_reviewers table OR extend claim_assignments with a REVIEWER role



1.3 Duplicate/Conflicting Table Definitions



Documents table: A2 schema has ocr_status ENUM with COMPLETED, Framework has DONE — pick one

Documents table: A2 has ocr_assignment_status, Framework has match_status — different names, same concept

Events table: A2 calls it events, Framework calls it timeline_events — different field sets

Folders table: A2 has document_folders with folder_type ENUM, Framework section E has folders with folder_type VARCHAR

Offer/Invoice: A2 has separate offers and invoices tables, Framework section C merges them as OFFER/INVOICE single entity

Fix: Use A2 schema as base (it's more complete SQL), align naming to Framework where clearer



1.4 Status ENUM Conflicts



A2: phase_status includes NICHT_BEGONNEN as initial state

Framework: Status machine starts at TERMIN_OFFEN

Fix: Keep NICHT_BEGONNEN — it's the pre-activation state before a phase starts



1.5 Priority ENUM Difference



A1/A2: NIEDRIG, NORMAL, HOCH, KRITISCH (German)

Framework: NORMAL, HIGH, CRITICAL (English, missing LOW)

Fix: Use German ENUMs from A2 (consistency with domain language)



1.6 DocumentType ENUM Difference



A2: 7 types (FOTO, PROTOKOLL, ANGEBOT, RECHNUNG, GUTACHTEN, KORRESPONDENZ, SONSTIGE)

Framework: 13 types (includes phase-specific: LO_PROTOKOLL, LB_PROTOKOLL, TROCKNUNGSPROTOKOLL, etc.)

Fix: Use Framework's richer ENUM (it's needed for OCR auto-classification)



1.7 Missing Tables in A2



ocr_extractions — referenced in Framework but not in A2 DDL

match_decisions — referenced in Framework but not in A2 DDL

notification_preferences — referenced in Framework section I but not in A2

Fix: Add these during schema creation



1.8 ERP Patch v1 Role Model vs Everything Else



ERP Patch v1 introduces 11 operational roles (AZUBI, BAUHELFER, FACHARBEITER, VORARBEITER, BAULEITER, PROJEKTLEITER, SCHADENMANAGER, LOGISTIK, EINKAUF, HR, GESCHAEFTSLEITUNG)

These are completely separate from the Schadenradar portal roles

Fix: Two separate role systems — Schadenradar portal roles for external users, ERP operational roles for internal users. Unified users table with a system field to distinguish context.

6. Proposed CLAUDE.md

This file goes in the repo root and instructs Claude Code on project conventions:

markdown# SüdWest Sanierung — Schadenradar + ERP



## Project Overview

Multi-tenant SaaS platform (Schadenradar) + internal ERP for a German property-damage restoration company. Built with Next.js 14, Supabase, tRPC, Shadcn/UI, TailwindCSS.



## Tech Stack

- **Monorepo:** Turborepo

- **Frontend:** Next.js 14 (App Router), React 18, TypeScript strict

- **UI:** Shadcn/UI + TailwindCSS (design tokens in tailwind.config.ts)

- **Backend:** Supabase (PostgreSQL, Auth, Storage, Realtime, Edge Functions)

- **API Layer:** tRPC v11 (end-to-end type safety)

- **Automation:** n8n (self-hosted or cloud)

- **OCR:** Azure Document Intelligence

- **Email:** Resend

- **Deployment:** Vercel (EU) + Supabase (EU Frankfurt)

- **Testing:** Vitest (unit), Playwright (E2E)



## Repository Structure

apps/

portal/          # Schadenradar (external SaaS portal)

erp/             # SWS ERP (internal operations)

admin/           # System administration

packages/

db/              # Database schema, migrations, seed

auth/            # Auth helpers, RBAC

types/           # Shared TypeScript types, Zod schemas

ui/              # Shared UI components

trpc/            # Shared tRPC routers



## Coding Conventions

- Language: TypeScript strict mode, no `any`

- UI text: German (this is a German-market product)

- Code/variables/comments: English

- ENUM values: German SCREAMING_SNAKE_CASE (e.g., `WASSERSCHADEN`, `TERMIN_OFFEN`)

- Database: snake_case for columns, PascalCase for ENUMs

- Components: PascalCase files, named exports

- Use Zod for all input validation

- Use `TIMESTAMPTZ` (not `TIMESTAMP`) for all time fields

- Every table gets `created_at` and `updated_at` with trigger

- Every tenant-scoped table gets redundant `tenant_id` for RLS performance



## Role Systems

### Schadenradar Portal Roles

SYSTEM_ADMIN, TENANT_ADMIN, MEMBER, VIEWER, REVIEWER, INTEGRATION



### ERP Internal Roles

GESCHAEFTSLEITUNG, PROJEKTLEITER, SCHADENMANAGER, BAULEITER, VORARBEITER, FACHARBEITER, BAUHELFER, AZUBI, LOGISTIK, EINKAUF, HR, CONTROLLING, FIB



## Key Business Rules

- ERP is Source of Truth for all operational data

- Schadenradar is a VIEW layer — it reads from ERP, writes only comments/documents/claims

- 5 damage phases: Leckortung → Leckagebehebung → Rückbau → Trocknung → Wiederherstellung

- Each phase follows the same 10-step status machine (NICHT_BEGONNEN through ABGESCHLOSSEN)

- Traffic light: GREEN (on track), YELLOW (at risk), RED (overdue/blocked)

- Multi-tenant: Multiplikatoren see ONLY their own data (enforced via Supabase RLS)

- No prices visible to roles below PROJEKTLEITER

- All state changes require audit trail

- No-Silent-Failure: every automation must log success or create an error task



## Design System

- Primary: #84bc47 (SWS Green)

- Gray: #6f6f6f

- Background: #f2f0ec

- Font: Inter

- Light mode only (no dark mode)

- Cards: rounded-lg, shadow-sm

- Sidebar: 256px fixed



## Before Each Session

1. Run `pnpm install` if packages changed

2. Check `packages/db/migrations/` for pending migrations

3. Run `pnpm dev` to start all apps

4. Run `pnpm typecheck` before committing



## Testing

- Unit tests: `pnpm test`

- E2E tests: `pnpm e2e`

- Type check: `pnpm typecheck`

- Lint: `pnpm lint`
